왜 Spring으로 전환하는가

약 5년간 PHP/Laravel로 커머스, 어드민, API 서버를 개발하며 백엔드의 핵심 역량(DB 설계, API 설계, 캐싱 전략, 비동기 처리)을 쌓았습니다. Laravel의 생산성은 충분히 활용했지만, 대규모 트래픽 처리와 엔터프라이즈 환경에서 요구하는 타입 안정성, 메시지 기반 아키텍처, 관측성(Observability) 인프라를 보다 체계적으로 다루기 위해 Spring 생태계로 전환을 결정했습니다.

PHP / Laravel → Java / Spring Boot

백엔드 핵심 역량은 유지 — 프레임워크와 생태계를 전환

이 프로젝트로 증명하고 싶은 것

  • 러닝 커브 돌파력 — Spring Boot 4.0 (최신 버전)을 선택하여, 안정적인 3.x가 아닌 변화가 많은 최신 생태계에서도 문제를 해결할 수 있음을 증명
  • 설계 역량의 이전 — Laravel에서 익힌 큐 처리, 캐싱, API 설계 패턴을 Spring 생태계의 도구(RabbitMQ, Redis, Spring Data)로 자연스럽게 재구현
  • 프로덕션 수준의 완성도 — 단순 CRUD가 아닌, 메시지 파이프라인 + 실패 관리 + 모니터링 + 테스트까지 포함한 풀 사이클 구현

2. 프로젝트 개요

OrderBridge란?

여러 쇼핑몰 채널(쿠팡, 지마켓, 옥션 등)에서 들어오는 주문을 단일 파이프라인으로 수집하고 정규화하여, WMS(창고관리시스템)까지 자동으로 전달하는 주문 관리 플랫폼입니다.

데이터 흐름

채널 API (쿠팡 / 지마켓 / 옥션) | v @Scheduled 60초 주기 폴링 Collector (채널별 수집기) | v Redis Set으로 중복 수집 방지 OrderCollectService (정규화 + 저장) | v RabbitMQ 발행 OrderProducer --> orders.exchange --> orders.inbound (Queue) | v Redis SETNX 멱등성 보장 OrderConsumer (처리) | +-----+-----+ | | v v WmsService SSE 실시간 알림 (재시도 포함) (관리페이지) | 실패 시 --> orders.dlx --> orders.dlq (재처리 API)

3. 핵심 기술 의사결정

3-1. Spring Boot 4.0 선택

선택지장점단점
Spring Boot 3.x 안정적, 레퍼런스 풍부 곧 메인스트림에서 밀림
Spring Boot 4.0 (선택) 최신 패러다임, Java 21 완전 지원 패키지 이전 다수, 레퍼런스 부족
선택 이유: 안정적인 3.x를 고르면 학습은 편하지만, 실무에서 마이그레이션을 만났을 때 대응 능력을 보여줄 수 없습니다. 4.0의 breaking change(패키지 이동, API 변경)를 직접 해결하면서 공식 문서와 소스 코드 기반 문제 해결 능력을 쌓았습니다.

3-2. RabbitMQ 메시지 파이프라인

선택지장점단점
동기 처리 (직접 호출) 단순 구현 채널 증가 시 병목, 장애 전파
Kafka 대용량 스트리밍 이 규모에 과도한 인프라
RabbitMQ (선택) 라우팅 유연, DLQ 네이티브 지원 Kafka 대비 처리량 한계
선택 이유: 주문 수집 → WMS 전달이라는 흐름은 "정확한 1회 전달"이 핵심이지, 대용량 스트리밍이 아닙니다. RabbitMQ의 Exchange → Queue 라우팅과 DLX/DLQ 패턴이 실패 관리에 최적이었습니다.

3-3. Redis 3중 활용 전략

Redis를 단일 캐시가 아닌, 세 가지 역할로 분리 활용했습니다:

역할자료구조적용 위치목적
조회 캐싱 String (JSON) @Cacheable / @CacheEvict DB 부하 감소 (TTL 10분)
중복 수집 방지 Set (SADD) OrderCollectService 같은 주문 재수집 차단
메시지 멱등성 String (SETNX) OrderConsumer 동일 메시지 중복 처리 차단
Laravel 경험과의 연결: Laravel에서도 Cache, Queue의 중복 방지(unique job)를 Redis로 처리했습니다. 같은 개념을 Spring의 @Cacheable, RedisTemplate.opsForSet(), setIfAbsent()로 재구현하면서 프레임워크는 바뀌어도 설계 패턴은 동일함을 체감했습니다.

4. 기술 Deep Dive

4-1. DLQ (Dead Letter Queue) 실패 관리

프로덕션 환경에서 메시지 처리 실패는 반드시 발생합니다. "실패한 메시지를 어떻게 관리하느냐"가 시스템의 신뢰성을 결정합니다.

구현 흐름

메시지 처리 실패 │ ▼ Spring AMQP 자동 재시도 (max-retries: 3) 3회 실패 → reject │ ▼ RabbitMQ가 DLX(Dead Letter Exchange)로 전달 orders.dlx ──→ orders.dlq (Dead Letter Queue) │ ▼ 관리자가 REST API로 재처리 요청 POST /api/v1/dlq/retry → DlqService.retryAll() │ ▼ DLQ에서 꺼내 원래 Exchange로 재발행 orders.exchange ──→ orders.inbound (정상 큐)

설계 포인트

  • 자동 재시도와 수동 재처리의 분리 — 일시적 오류는 3회 재시도로 자동 복구, 지속적 오류는 DLQ에 격리 후 원인 파악 뒤 수동 재처리
  • 메시지 유실 방지 — default-requeue-rejected: false로 무한 재시도 루프를 방지하면서도, DLQ로 이동시켜 메시지를 보존
  • 운영 API 제공 — GET /api/v1/dlq/count로 현황 파악, POST /api/v1/dlq/retry로 일괄 재처리

4-2. 클레임 다형성 설계 (CTI)

취소/반품/교환은 공통 속성(대상 상품, 사유, 상태)과 타입별 고유 속성(환불금액, 수거지, 교환상품코드)이 혼재합니다.

JPA 상속 전략 선택

전략특징판단
SINGLE_TABLE 하나의 테이블에 모든 컬럼 NULL 컬럼 다수, 데이터 정합성 약화
JOINED (선택) 공통 + 타입별 테이블 분리 정규화 유지, 확장에 유리
TABLE_PER_CLASS 타입별 완전 분리 공통 쿼리 비효율 (UNION)
@Entity @Inheritance(strategy = InheritanceType.JOINED) @DiscriminatorColumn(name = "claim_type") public abstract class Claim { public abstract ItemStatus getRequestedItemStatus(); // 타입별 다른 상태 반환 public abstract ItemStatus getApprovedItemStatus(); // 다형성으로 분기 제거 }

4-3. Observability (관측성)

Micrometer로 비즈니스 메트릭을 수집하고, Prometheus + Grafana로 시각화합니다.

메트릭타입용도
order.collected.count Counter (채널별 태그) 채널별 주문 수집량 추이
wms.delivery.count Counter (성공/실패 태그) WMS 전달 성공률 모니터링
order.processing.time Timer 주문 처리 지연 시간 (p50, p95, p99)
설계 판단: Timer는 Consumer의 전체 handle()이 아닌, 중복 체크 이후 실제 비즈니스 로직만 감싸도록 배치했습니다. 중복 메시지의 빠른 반환이 평균 처리 시간을 왜곡하는 것을 방지하기 위함입니다.

5. 트러블슈팅

Spring Boot 4.0은 레퍼런스가 제한적입니다. 아래는 실제로 마주치고 해결한 주요 이슈들입니다.

1. Spring Boot 4.0 패키지 이전으로 인한 컴파일 에러
원인: @WebMvcTest가 org.springframework.boot.test.autoconfigure.web.servlet에서 org.springframework.boot.webmvc.test.autoconfigure로 이동. @MockBean이 제거되고 @MockitoBean으로 대체.
해결: Spring Boot 4.0 마이그레이션 가이드와 실제 JAR 내 패키지 구조를 확인하여 import 경로 수정. 3.x 기반 StackOverflow 답변에 의존하지 않고 공식 소스 기반으로 해결.
2. @WebMvcTest에서 JPA metamodel must not be empty
원인: @EnableJpaAuditing이 메인 Application 클래스에 위치하여, @WebMvcTest 슬라이싱 시에도 JPA Auditing이 로드되지만 엔티티가 없어 실패.
해결: @EnableJpaAuditing을 별도 @Configuration 클래스로 분리. 웹 슬라이스 테스트에서는 자동으로 제외되고, 풀 컨텍스트에서는 컴포넌트 스캔으로 정상 로드.
3. Testcontainers 2.x artifact 호환성 문제
원인: Testcontainers 2.x에서 artifact 이름이 변경됨 (postgresql → testcontainers-postgresql). PostgreSQLContainer<?> 제네릭 타입 파라미터 제거. junit-jupiter 모듈 삭제.
해결: build.gradle의 실제 의존성 버전을 먼저 확인한 뒤, Maven Central에서 2.x 호환 artifact를 검증하여 적용. 교훈: 의존성 추가 시 반드시 프로젝트 실제 버전 확인 후 진행.
4. Prometheus 메트릭 No Data (WMS/Processing)
원인: Micrometer Counter/Timer는 첫 increment()/record() 호출 시점에 등록됨. 이벤트 발생 전까지 Prometheus에 메트릭 자체가 노출되지 않아 Grafana에서 No Data.
해결: 생성자에서 메트릭을 사전 등록(pre-registration)하여, 이벤트 발생 전에도 0값으로 Prometheus에 노출되도록 처리.
5. DLQ 메시지가 쌓이지 않는 문제
원인: Consumer에서 의도적으로 throw하는 테스트 코드가 Redis 중복 체크(SETNX) 이후에 위치하여, 두 번째 시도부터는 중복으로 판단되어 예외가 발생하지 않음.
해결: 강제 실패 코드를 중복 체크 이전으로 이동하여 매 재시도마다 실패가 발생하도록 수정. Spring AMQP의 max-retries: 3 소진 후 DLQ로 정상 이동 확인.

8. Laravel 경험이 Spring에 어떻게 연결되는가

기술 스택은 바뀌었지만, 백엔드 엔지니어링의 핵심 개념은 동일합니다. 아래는 Laravel에서 쌓은 경험이 이 프로젝트에서 어떻게 적용되었는지 보여줍니다.

개념Laravel 경험Spring 적용
큐 / 비동기 Laravel Queue (Redis/SQS 드라이버), Job dispatch, Failed Jobs 테이블 RabbitMQ + Spring AMQP, DLQ/DLX 패턴으로 실패 관리
캐싱 Cache::remember(), Cache::tags(), Redis 드라이버 @Cacheable/@CacheEvict + RedisCacheManager, TTL 설정
ORM 설계 Eloquent 관계, Polymorphic Relations JPA @ManyToOne, @Inheritance(JOINED) CTI 전략
마이그레이션 Laravel Migration (up/down) Flyway 버전 관리 (V1~V7)
예외 처리 App\Exceptions\Handler, render() @ControllerAdvice/@RestControllerAdvice 분리, ErrorCode enum
스케줄링 Task Scheduling (Kernel.php) @Scheduled + Strategy 패턴 (채널별 Collector)
API 설계 Laravel API Resources, Route::apiResource @RestController + DTO 응답 분리, Swagger 자동 생성
인증 Laravel Auth, Middleware, Guard Spring Security FilterChain, SecurityConfig
핵심 메시지: Laravel 4년 9개월의 경험은 "PHP만 할 줄 안다"가 아니라, 큐, 캐싱, ORM, 예외 처리, 인프라 등 백엔드 엔지니어링의 본질을 이해하고 있다는 의미입니다. Spring은 이 본질을 더 큰 규모로 적용할 수 있는 도구이며, 이 프로젝트는 그 전환이 자연스럽게 이루어졌음을 보여줍니다.
